# 테스트 컨벤션

이 프로젝트는 Vitest를 사용해 유닛 테스트와 E2E 테스트를 분리한다. 테스트는 실행 속도와 검증 범위를 기준으로 먼저 유닛 테스트로 작성하고, Nest 애플리케이션 부팅, 라우팅, 모듈 wiring, 실제 HTTP 응답까지 확인해야 할 때만 E2E 테스트로 작성한다.

## 공통 규칙

- 테스트 파일은 `test/` 아래에 둔다.
- 테스트 대상과 가까운 하위 디렉터리를 사용한다. 예: `test/metrics/metrics.service.spec.ts`
- `describe()`에는 테스트 대상 이름을 적고, `it()`에는 조건과 기대 결과가 드러나는 한국어 문장을 적는다.
- 하나의 `it()`은 하나의 작업 단위를 호출하고, 하나의 특정 종료점을 검증 목표로 삼는다.
- 같은 실행 결과를 확인하기 위한 상태 코드, body, header 검증은 하나의 `it()`에 함께 둔다.
- 성공, 실패, 예외, 경계값, 권한, validation처럼 실행 경로나 기대 결과가 달라지면 `it()`을 분리한다.
- 비동기 동작은 `async/await` 또는 Vitest의 `resolves`/`rejects` matcher로 명확히 검증한다.
- 테스트 간 상태를 공유하지 않는다. 공유 리소스가 필요하면 `beforeEach`에서 만들고 `afterEach`에서 정리한다.
- 테스트는 같은 조건에서 항상 같은 결과를 내야 한다.

## 유닛 테스트

- 파일명은 `*.spec.ts`를 사용한다.
- 실행 명령은 `pnpm test`이다.
- 순수 서비스, 함수, 작은 단위의 비즈니스 로직을 대상으로 한다.
- HTTP 서버, 실제 Nest 애플리케이션 부팅, 외부 I/O는 사용하지 않는다.
- 필요한 의존성은 직접 생성하거나 가벼운 mock/stub으로 대체한다.
- Nest testing module은 DI 구성 검증이 필요한 경우에만 사용한다.
- 작업 단위는 진입점 호출부터 눈에 띄는 결과가 나타나는 종료점까지의 흐름이다.
- 진입점은 보통 public 메서드나 함수 호출이고, 종료점은 반환값, 예외, 상태 변경, 의존성 호출 중 하나다.
- 바깥 `describe()`는 테스트 대상 단위로 작성한다. 예: `describe('MetricsService')`
- 안쪽 `describe()`는 메서드나 주요 public API 단위로 작성한다. 예: `describe('getMetrics')`
- `it()`은 해당 메서드의 조건과 하나의 종료점 단위로 작성한다. 예: `it('metrics가 수집되면 Prometheus metrics를 반환한다')`
- 반환값, 예외, 상태 변경, 의존성 호출은 종료점 종류가 다르므로 각각 별도 `it()`으로 분리한다.
- 의존성은 단위 테스트 중 온전히 제어할 수 없는 파일 시스템, 네트워크, DB, 외부 팀 코드, 오래 걸리는 계산을 의미한다.
- 의존성이 필요하면 stub으로 대체하되, 의존성 호출 자체를 검증하는 테스트는 최소화한다.

## E2E 테스트

- 파일명은 `*.e2e-spec.ts`를 사용한다.
- 실행 명령은 `pnpm test:e2e`이다.
- Nest 애플리케이션 전체를 띄워 라우팅, 모듈 wiring, 컨트롤러 응답을 검증한다.
- 실제 네트워크, REST API, 시스템 시간, 파일 시스템, DB처럼 단위 테스트에서 제어하기 어려운 요소를 사용하면 E2E 또는 통합 테스트로 분리한다.
- E2E 테스트는 단위 테스트에서 검증하지 못한 시스템 상호 작용을 확인하는 용도로 작성한다.
- Fastify 기반이므로 `createE2eApp()`과 `app.inject()`를 표준으로 사용한다.
- 각 테스트 파일은 `beforeEach`에서 앱을 만들고 `afterEach`에서 `app.close()`로 종료한다.
- 바깥 `describe()`는 컨트롤러 단위로 작성한다. 예: `describe('MetricsController (e2e)')`
- 안쪽 `describe()`는 컨트롤러 메서드와 라우트 단위로 작성한다. 예: `describe('GET /metrics')`
- `it()`은 해당 라우트의 요청 조건과 기대 응답 단위로 작성한다. 예: `it('정상 요청이면 Prometheus metrics를 반환한다')`
- 정상 응답, 잘못된 요청, 인증/인가 실패, 리소스 없음, 서버 오류처럼 HTTP 결과가 달라지면 `it()`을 분리한다.
- HTTP 상태 코드, 응답 body, 주요 header를 함께 검증한다.

구조 체크리스트:

- [ ] 파일명이 `*.e2e-spec.ts`인가?
- [ ] 바깥 `describe()`가 컨트롤러 이름인가?
- [ ] 안쪽 `describe()`가 `GET /metrics`처럼 HTTP method와 route를 함께 나타내는가?
- [ ] 각 `it()`이 하나의 요청 조건과 기대 응답만 검증하고, HTTP 결과가 다르면 분리했는가?
- [ ] 같은 응답을 설명하는 status code, body, header 검증은 하나의 `it()`에 함께 있는가?
- [ ] `createE2eApp()`으로 앱을 만들고 `app.inject()`로 요청한 뒤 테스트마다 앱을 닫는가?

## 실행 명령

```bash
pnpm test        # 유닛 테스트
pnpm test:e2e    # E2E 테스트
pnpm test:watch  # 유닛 테스트 watch 모드
pnpm test:cov    # 유닛 테스트 커버리지
```

PR 전에는 변경 범위에 맞는 테스트를 실행한다. 서비스나 함수만 변경했다면 `pnpm test`를 실행하고, 라우트, 모듈 구성, 애플리케이션 부팅 흐름을 변경했다면 `pnpm test:e2e`도 함께 실행한다.
